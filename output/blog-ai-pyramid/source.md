## 问题的起点

我日常在 tmux 里同时开好几个 Claude Code，每个窗口一条任务线，互不干扰。大部分时候效率很高。

但问题出在第四件事上。**同时跟三条线还行，第四条就容易忘掉。** 忙着前三件事的时候，第四个窗口里 Claude Code 早就干完了在等我，或者卡在一个权限确认上等我点"允许"，我完全没注意到。等我想起来切过去看，已经过了二十分钟。

所以我开始想：**能不能让一个 AI agent 去操控另一个正在运行的 AI agent？** 让一个调度者 agent 替我看着就行。

具体方案：在 OpenClaw 上跑一个常驻的调度 agent，通过 Discord 跟我对话。但这一层更像是一个对外的壳——换成飞书 bot、Slack bot 都行。真正难的是壳下面的问题：**调度者怎么接入我已经在跑的 Claude Code session？**

它是一个独立的 agent，有自己的 tool call，但这些工具操作的是它自己的环境。我 tmux 里正在跑的那几个 Claude Code？它看不到，也操控不了。

---

## 执行者选谁？为什么是 Claude Code

"AI 操控 AI"不是新概念。首先要回答：**执行者选谁？**

调度者可以是任何能发指令的 agent，但执行写代码的那个 agent 选什么？OpenClaw 的 agent 底层也是 Claude 模型，模型能力没差异，但它运行 agent 时用的是一个 Python 开源框架，不是 Claude Code 本身。

实际试用后发现差距在产品层面——Claude Code 有子进程调度、Plan Mode、Explore Agent 这些更成熟的功能。**选 Claude Code 不是因为模型更聪明，而是因为它作为写代码的产品更完善。**

确定了"下发给 Claude Code"之后，问题变成：**怎么下发？**

---

## 调研：五种方案为什么都不够用

我花了大量时间调研，所有方案都倒在同一个问题上。

**Anthropic Agent SDK** — 官方 SDK，`query()` 本质是 spawn 新的 Claude Code 子进程，通过 stdin/stdout 做 JSON 通信。能拿到结构化输出，但**只能创建新会话，不能接入已经在跑的 session**。`resume` 功能也不是连接运行中的进程——是从磁盘加载历史、spawn 新进程重放上下文。

**ACP 协议** — 由 Zed 和 JetBrains 推动的 agent 间通信标准，acpx 是它的无头 CLI 客户端，支持 13+ 种 coding agent。设计很有远见，但底层和 Agent SDK 一样：通过适配器 spawn 新进程。

**Agora 框架** — 基于 tmux 的多 Agent 编排框架，投票、审批、状态机、Gate 门禁，三层架构解耦很干净。但面向民主协作场景，我需要的是金字塔式的"老板下指令，员工去干"。而且它的 tmux 层只认自己创建的 pane，不能接管已有 session。

---

## 调研（续）：剩下的方案也不行

**Claude Code Remote Control** — 官方远程操控功能，能接入已有 session。但必须走 Anthropic 的云中继，不能自建，断网就没了。

**社区 Web UI** — claude-code-web、claude-code-webui 等项目，全是新建 session。

**所有方案都倒在同一个问题上：我 tmux 里已经开着好几个 Claude Code，上下文全在里面，新建进程意味着丢掉所有上下文。**

---

## 最终解法：tmux send-keys + JSONL

调研到最后发现，最直接的方案反而最简单：

```bash
# 给正在运行的 Claude Code 发指令
tmux send-keys -t %3 "fix the login bug" Enter
```

加上 Claude Code 的 JSONL session 文件可以读到结构化的对话历史——两条通道一组合：

- **输入走 tmux** — `send-keys` 给正在运行的 Claude Code 发任何文本
- **输出读 JSONL** — 从 `~/.claude/projects/*/session.jsonl` 读结构化结果

就是一个能 attach 已有 session、又能拿到结构化数据的完整方案。不需要框架，不需要 SDK，几十行脚本就能跑。

我把这个做成了开源项目 **AImux**。

## 意外收获：涌现智能

本来只是为了解决"干活时不能聊天"的问题。但用了一段时间后发现了一个比预期有意思得多的现象：

**下层 AI 会干出上层 AI 没想到的事。**

真实案例——我需要打通"任务完成后自动通知"这条链路，给 Claude Code 的指令只有一句："看看 system event 为什么不工作。"

它自己干了什么：

1. 读平台源码 → 发现命令走 WebSocket
2. 跑命令 → ws 握手超时
3. 追代码 → Node.js event loop 被阻塞了 3 秒
4. 继续追 → 定位到**飞书插件的 jiti 编译器**卡住了整个进程
5. 建议禁用飞书插件 → 修复

**我从来没提过飞书。** 它通过一层层源码分析自主找到了根因。**整个排查跨越 4 种方案，零人工干预。**

---

## 涌现智能（续）：故事没结束

修完飞书插件的 bug 后发现 system event 还是不好使——平台把触发事件的 reason 字段硬编码成了 "wake"，AI 收到的消息是一句毫无信息量的"如果没事就回 OK"。

于是它又自主搜索，找到另一个可行命令，验证通过，还顺手提了 3 个 issue。

从飞书插件到 reason 字段，**两轮完全不同的排查方向，都是 agent 自主发起的。**

---

## 涌现的四个前提条件

这不是 prompt engineering 的结果，没有人在指令里写"检查插件是否阻塞 event loop"。而是执行者在过程中根据中间发现自主调整方向。

为什么只有在"下发"模式下才会出现？

**1. 完整工具链** — Claude Code 能读任意文件、跑命令、搜代码，调查链路不会断。

**2. 足够的自主空间** — 不是"帮我查第 42 行"，而是"看看为什么不工作"。指令越模糊，agent 越可能走出意想不到的路径。

**3. 中间反馈闭环** — 每一步的报错自动成为下一步的线索。ws 超时 → 追 event loop → 发现 jiti → 禁用飞书，这条链是报错驱动的。

**4. 不被打断** — 后台执行，没人催进度，agent 可以沉浸式追踪。

如果上层 agent 自己做，它一边跟你聊天一边干活，每个操作阻塞对话，根本没法深入追踪。**分层本身就是涌现的前提。**

---

## 背后的关键：上下文管理

做 agent 开发有很多重要的事，但做下来我越来越觉得，**上下文管理是最核心的一件**。

分层之所以有效，本质上是因为它让每一层的上下文保持精简。调度者不需要知道执行者读了哪些文件、试了哪些命令、中间踩了什么坑——它只需要对两件事负责：**发出去的指令**和**收回来的结果**。中间的几十轮工具调用、几百行代码 diff，全部留在执行者自己的上下文里，不会污染调度者的窗口。

反过来，如果调度者把所有执行细节都吞进自己的上下文，它很快就会被细节淹没，做出的判断也会变差——该汇报的忘了汇报，该派活的自己动手干，正经事反而被挤掉了。

> 一个靠谱的负责人不会坐在每个工位后面盯着代码写，他只关心"这件事交给谁了"和"结果怎么样"。**信息隔离不是偷懒，是让每个角色都能在自己的上下文密度下最好地工作。**

## 一个新认知：AI 没有意志力

搭这套系统踩过最大的坑不是技术问题，是认知问题。

我的调度器 agent 每次启动是全新 session。你告诉它"下次记得派活而不是自己做"，它回"好的我记住了"——然后下次启动，什么都不记得。

**AI agent 没有意志力，只有机制。**

人类可以说"下次注意"然后靠自制力改行为。AI 不行——如果一个规则没写在配置文件里，它就不存在。

**"下次注意"是人类的解法，"写入文件"才是 agent 的解法。**

---

## 写入文件：具体怎么做

所以我把所有运行逻辑写进了启动必读文件：

```markdown
# AGENTS.md（每次启动第一时间读）

你是调度器，不是执行者。

绝不做的事：
- 不要自己写代码——下发给 Claude Code
- 不要空头承诺"下次注意"——改文件改配置
- 不要反复轮询状态——用后台监听
```

这段话不是建议，是**运行时加载的系统指令**。你想让 agent 改行为，别说，写。

## 大家都在做，但没有人做的一样

做 AImux 之前我调研了很多类似实现。**ACPX** 是比较权威的一个——Zed 和 JetBrains 推动的官方协议客户端，设计很完整。

另一方面，我把这个想法发到微信讨论群之后，很多人都分享了他们自己做的类似东西。大家对"AI 调度 AI"这种架构其实都有想法，而且直接就去做了。

但我看了他们的实现后发现，**和我的并不完全重合**。差异不在通信方式，而在运行模式和协作结构——有人启动长驻进程持续监听，有人每次跑一个短任务就退出；有人用聊天 API 让 agent 之间对话协商，有人由一个 leader 统一分配任务。架构范式完全不同。于是我也去做了自己的版本。

以前遇到这种事，第一反应一定是"先找别人做好的"，因为自己从头做太麻烦了。但现在反而是：**自己做最符合自己的场景，而且从 idea 到能用的速度非常快。** 从发现问题到写完代码到验证可行，前后也就几天。

## 想跟大家聊的

这套"AI 操控 AI"的玩法还在早期，几个我自己也没想清楚的问题：

- **多层 agent 的信任边界怎么划？** 上层给下层多大的自主权？完全放手 vs 每步审批，中间有没有更好的平衡点？
- **涌现智能能被稳定复现吗？** 还是说它本质上是概率性的，只能创造条件然后等它出现？
- **你们有类似的实践吗？** 不一定是 tmux，任何"一个 AI 调度另一个 AI"的经验都想听。
